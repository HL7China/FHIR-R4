1.9 Getting Started with FHIR 开始使用FHIR

工作组:FHIR Infrastructure 成熟度: N/A标准状态: Informative

FHIR is a platform specification that defines a set of capabilities use across the healthcare process, in all jurisdictions, and in lots of different contexts. While the basics of the FHIR specification are relatively straight-forward (see the Overviews: General, Developers, Clinical, and Architects), it can still be difficult to know where to start when implementing a solution based on FHIR. FHIR是一种平台规范，定义了整个医疗保健流程，所有司法管辖区以及许多不同环境中使用的一组功能。虽然FHIR规范的基础知识相对简单 （参见概述：一般， 开发人员，临床和架构师），但在实现基于FHIR的解决方案时，仍然很难知道从哪里开始。

This page provides some guidance to help get new implementers started on their path to successful implementation. Beyond reading the overviews (previous paragraph), where should an implementer start? Generally, an implementer needs to resolve: 本页提供了一些指导，以帮助新的实现者开始成功实施。除了阅读概述（上一段），实施者应该从哪里开始？通常，实施者需要解决：

  • How will information be exchanged? (see Foundation Module 如何交换信息？（详见基础组件）
  • How are terminologies being used? (see Terminology Module) 如何使用术语？（详见术语组件）
  • How will the information be secured? (see Security and Privacy Module) 如何保护信息？（详见安全和隐私组件）
  • When is information exchanged? (See Workflow Module) 什么时候交换信息？（详见工作流程组件）
  • What information is going to be exchanged? 要交换什么信息？

The remaining sections provide guidance on specific areas (Foundation, Implementer Support, Security and Privacy, Conformance, Terminology, Linked Data, Administration, Clinical, Diagnostics, Medications, Workflow, Financial and Clinical Reasoning). 剩余部分提供关于特定领域的指导（基础，实施者支持，安全和隐私，一致性，术语，关联数据，管理，临床，诊断，药物，工作流程，财务和临床推理）。

All implementers should be aware of how versioning works in the FHIR specification. See both: 所有实施者都应该知道版本控制在FHIR规范中的工作原理。见两个：

  • Managing Multiple FHIR Versions管理多个fhir版本
  • Specification Version Management Policy 规范版本管理策略

1.9.1 Modules组件

In order to help implementers find their way around the specification and answer these questions, it is organized into a set of "modules". Each module represents a different functional area of the specification, and contains: 为了帮助实施者找到解决方案并回答这些问题，它被组织成一组“组件”。每个组件代表规范的不同功能区域，包含：

  • Scope and Index: A description of the content covered by the module, and an index of the important content 范围和索引：对组件所涵盖内容的描述，以及重要内容的索引。
  • Use cases: Guidance for common uses of the module, and how to approach them. This is a key resource for implementers familiarizing themselves with the FHIR specification 用例：组件的常用用法指南，以及如何处理它们。这是熟悉FHIR规范的实施者的关键资源
  • Security / Privacy: Information安全/隐私：信息
  • Roadmap: Where the content covered by the module is in terms of overall progress (see also, for general information: FHIR Timelines)路线图：组件涵盖的内容在整体进度方面（另请参见一般信息：FHIR时间表）

Broadly, the modules are organized into 3 groups: 从广义上讲，这些组件分为3组：

  • Infrastructure (bottom rung, and bottom row of boxes)基础设施（底层阶梯和底层框架）
  • Content (middle rung, and top row of boxes)内容（中间阶梯和顶层框架）
  • Reasoning (top rung)推理（最顶层）

Dependencies between the modules are mainly downwards, with some horizontal dependencies. Implementers should choose the content modules to engage with based on their requirements, and should only engage with the reasoning module if they need to do clinical decision support, and/or Quality Measures.
 组件之间的依赖关系主要是向下的，具有一些水平依赖性。实施者应根据自己的要求选择要参与的内容组件， 并且只有在需要做出临床决策支持和/或质量措施时才应与推理组件合作。

In addition to the use case based assistance in the modules, these additional documentation pages may be useful:除了在组件中提供基于用例的帮助外，这些附加的文档页面可能会很有用：

  • Common Use Cases: Personal Health Record, Document Sharing (XDS) and Decision Support 常见用例：个人健康记录、文档共享（xds）和决策支持
  • Resource Guide: Further information about the resources and the relationship between them 资源指南：有关资源及其关系的进一步信息

Finally, one important place to look is the registry of implementation guides , to see whether similar (or identical) requirements have been met.最后，要查看的一个重要地方是实现指南的注册，以查看是否满足了相似（或相同）的需求。
